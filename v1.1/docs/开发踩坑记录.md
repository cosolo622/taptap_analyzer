## 一、爬虫开发踩坑

### 1.1 API请求失败

**问题描述**:
尝试直接使用requests请求TapTap API获取评价数据。

```python
url = "https://www.taptap.cn/webapiv2/review/v2/by-app?app_id=258720"
response = requests.get(url)
# 返回404
```

**原因分析**:
- TapTap API需要登录态
- 需要特殊的headers（User-Agent、Referer等）
- 可能有反爬机制

**解决方案**:
改用Selenium模拟浏览器访问，绕过反爬检测。

### 1.2 星级解析错误（核心坑）

**问题描述**:
爬取的星级全部显示为5星，无法区分不同评分。

**调试过程**:

1. **初步分析**: 页面显示5个星星，检查HTML发现有10个SVG元素
   ```html
   <div class="review-rate__normal">  <!-- 灰色背景星星 -->
     <svg class="review-rate__star">...</svg> × 5
   </div>
   <div class="review-rate__highlight">  <!-- 实际评分星星 -->
     <svg class="review-rate__star">...</svg> × 5
   </div>
   ```

2. **错误方案**: 通过SVG的fill颜色判断星级
   ```python
   # 错误：所有评价的后5个星星都是青色
   bright_count = sum(1 for svg in star_svgs if 'rgb(0, 217, 197)' in fill)
   ```

3. **正确方案**: 从highlight容器的width属性计算星级
   ```html
   <div class="review-rate__highlight" style="width: 72px;">
   ```
   ```python
   # 正确：width / 18 = 星级
   width = int(re.search(r'width:\s*(\d+)px', style).group(1))
   rating = round(width / 18)  # 每个星星18px
   ```

**关键发现**:
- 用户提供的HTML片段揭示了真相：星级通过CSS的width属性控制显示
- 72px = 4星，90px = 5星，54px = 3星

### 1.3 日期解析错误

**问题描述**:
部分评价的日期显示为今天，但实际是几天前。

**调试过程**:

1. **初步分析**: 页面显示"20小时前"等相对时间
   ```html
   <span class="tap-time">20 小时前</span>
   ```

2. **错误方案**: 直接解析文本
   ```python
   # 错误：20小时前被解析为今天
   if '小时前' in time_text:
       review_date = datetime.now().strftime('%Y-%m-%d')
   ```

3. **正确方案**: 从title属性获取完整日期
   ```html
   <span class="tap-time" title="2026/02/26 21:48:26">20 小时前</span>
   ```
   ```python
   # 正确：从title获取完整日期
   title = time_elem.get_attribute('title')  # "2026/02/26 21:48:26"
   date_match = re.match(r'(\d{4})/(\d{2})/(\d{2})', title)
   ```

### 1.4 内容过滤问题

**问题描述**:
爬取的评价内容包含用户名、游戏时长等干扰信息。

**示例**:
```
不屈的钦王 玩过 作为一个从Steam端游时代就一路走来的老玩家...
```

**解决方案**:
```python
# 过滤用户名和游戏时长
content = re.sub(r'^[^\s]+\s*', '', content)  # 移除开头的用户名
content = re.sub(r'玩过\s*$', '', content)    # 移除结尾的"玩过"
```

## 二、情感分析踩坑

### 2.1 关键词匹配误判（核心坑）

**问题描述**:
纯正向评价被错误标记为负面问题。

**示例**:
```
评价内容：作为一个从Steam端游时代就一路走来的老玩家，说实话，当听说《鹅鸭杀》出手游的时候，我的内心是忐忑的...

错误结果：
- 情感：正向
- 问题分类：商业化问题-太氪金/收费高
- 关键词：贵、坑、版本、操作、bug、音效、角色、有意思
```

**原因分析**:
1. 关键词匹配过于宽泛
2. 提取的关键词与实际内容不符（评价中没有"贵"、"坑"、"bug"）
3. 情感和分类没有联动（正向评价不应该有负面标签）

**解决方案**:
使用GLM大模型直接分析，输出结构化结果：
```json
{
  "sentiment": "正向",
  "issues": [],
  "summary": "端游老玩家的情怀之作，原汁原味还原经典"
}
```

---

### 2.2 分类标签设计

**问题**:
固定分类标签无法覆盖所有问题类型。

**解决方案**:
让GLM灵活归纳问题类型，不限定固定标签：

| 大类 | 细分问题 |
|------|----------|
| 环境问题 | 低素质玩家、路人局体验差、语言攻击、村规问题、场外现象 |
| 技术问题 | 优化不足、网络波动、闪退、发热严重 |
| 商业化问题 | 氪金点过多、皮肤定价、福利少 |
| 平衡性问题 | 角色强度不均、阵营优势过大 |
| 内容问题 | 不耐玩、角色太少、地图太少 |
| 匹配问题 | 水平差距大、段位匹配不合理 |
| 功能建议 | 排位赛模式、单机模式、自定义皮肤 |

## 三、性能优化踩坑

### 3.1 滚动加载效率

**问题描述**:
每次滚动只加载10-15条评价，爬取200条需要很长时间。

**优化方案**:
```python
# 滚动前检查是否已达到目标数量
while len(reviews) < max_reviews and scroll_count < max_scrolls:
    driver.execute_script("window.scrollTo(0, document.body.scrollHeight);")
    time.sleep(2)  # 等待加载
    scroll_count += 1
```

### 3.2 内存优化

**问题描述**:
大量评价数据占用内存过多。

**解决方案**:
```python
# 分批保存到文件
batch_size = 20
for i in range(0, len(reviews), batch_size):
    batch = reviews[i:i+batch_size]
    with open(f'batch_{i//batch_size + 1}.json', 'w') as f:
        json.dump(batch, f)
```

## 五、v1.1 新增踩坑

### 5.1 Scrapling安装失败（2026-03-11）

**问题描述**:
安装 scrapling 或 playwright 时报错：
```
error: Microsoft Visual C++ 14.0 or greater is required.
```

**原因分析**:
- scrapling/playwright 依赖 greenlet>=3.1.1
- greenlet 需要编译 C 扩展
- Windows 上需要 Visual C++ Build Tools（约6GB）

**解决方案**:
继续使用 Selenium（已安装 v4.36.0），修复已知解析问题。

## 七、经验总结

1. **优先检查HTML结构**: 很多问题可以通过仔细检查HTML元素解决
2. **利用隐藏属性**: title、data-*等属性往往包含有用信息
3. **大模型更准确**: 对于情感分析等任务，大模型比传统方法更准确
4. **保留调试脚本**: 调试脚本记录了问题排查过程，有参考价值
5. **分批处理大数据**: 大量数据分批处理，避免内存溢出
6. **Windows编译问题**: 遇到C扩展编译失败，优先考虑预编译包或替代方案
