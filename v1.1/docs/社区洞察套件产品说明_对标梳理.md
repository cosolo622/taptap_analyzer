# 小红书舆情洞察模块产品设计方案（运营视角重构版）

## 1. 方案背景与核心理念
本方案服务于**社区运营负责人**，旨在提供对小红书平台的全面洞察，涵盖舆情监控、社区内容、趋势、新版本口碑、用户建议、竞品内容运营策略及KOL发现等。
基于用户的最新反馈，我们不再生搬硬套飞书原有的所有模块，而是**站在运营专家视角**，将有价值的数据深度挖掘并重新组织前端展现。

核心调整包括：
1. **清理无用模块**：移除“简介”、“使用手册”（移至右上角下拉菜单）、“分析”（被拆解为更具体的Tab）、“标签”、“空间管理”（重命名为产品配置）。
2. **精细化数据分类**：明确区分“水帖/水评论”与“有价值内容”，拆分主帖与高赞评论的展示逻辑。
3. **引入运营算法模型**：针对热点发现、KOL评估、风控预警引入具体的计算公式和阈值策略。
4. **多产品架构支持**：支持按具体游戏产品（甚至竞品）配置监控体系，不同产品的配置和数据互相隔离。

---

## 2. 小红书监控左侧菜单架构（最终确认版）

在 `App.vue` 的左侧菜单中，小红书监控模块将包含以下 Tab：

1. **实时舆情数据**：对标 TapTap 的实时大盘。
2. **词云分析**：对标 TapTap 的词云看板。
3. **情感和主题趋势**：新增的情感与分类趋势追踪。
4. **社区热点**：高热度话题自动聚类与分天趋势。
5. **每日摘要**：AI 生成的日/周/月报。
6. **社媒内容**：帖子与高赞评论的内容检索库。
7. **KOL发现**：基于互动指标的优质达人挖掘。
8. **风控预警**：基于跨天同环比的负面爆发监控。
9. **产品配置**：原“空间管理”，针对不同产品定制话题和关键词。
10. **数据更新**：独立于展示页的爬虫执行与调度中心。
11. **数据库管理**：支持底层 SQL 运维。

## 3. 各模块功能定义与算法策略

### 3.1 实时舆情数据
- **定位**：纯数据图表展示，移除爬虫开关（移至“数据更新”Tab）。
- **功能**：
  - 支持日期范围筛选查询。
  - 展示今日实时核心指标（总帖子数、总互动量、预估曝光量）。
  - 展示今日正负向占比圆环图、今日发帖时间分布折线图。

### 3.2 词云分析
- **定位**：高频词汇视觉化。
- **功能**：模仿 TapTap 的看板，可按天/周/月筛选，直观展示小红书语境下的高频正负向词汇。

### 3.3 情感和主题趋势
- **定位**：宏观舆情走向监控。
- **功能**：
  - **情感趋势**：按天展示正向、负向、中性帖子数量的折线堆叠图。
  - **主题趋势**：按天展示不同主题（如：玩法、商业化、Bug、活动）的声量占比柱状图。

### 3.4 社区热点
- **定位**：发现“今天什么样的话题成为了热点”。
- **功能**：
  - **按日查询**：选择日期，默认展示 Top 20 热点（最多支持 100 条）。
  - **热度计算模型**：
    - 根据小红书特性（收藏分享价值高），定义热度（声量）公式：`热度值 = (点赞数 * 1 + 评论数 * 1.5 + 收藏数 * 2 + 分享数 * 2.5) * 时效衰减系数`。
  - **话题指标展示**：每个热点话题展示包含的帖子总数、以及该话题下累计的转评赞藏总数（若有播放量/浏览量一并展示）。
  - **趋势下钻**：点击某个热点，可查看该话题在过去 N 天的热度趋势变化图。
  - **自定义话题体系**：由于每款游戏关注的话题不同（如A游戏看皮肤，B游戏看数值），热点话题的分类词库支持在“产品配置”中由用户自定义。

### 3.5 每日摘要
- **定位**：AI 驱动的自动化舆情报告。
- **功能**：
  - **自动生成**：每天 23:00 定时调用大模型 API 自动生成当日总结。
  - **手动生成**：支持手动选择某一天、某一周或某一月生成摘要（日报/周报/月报）。
  - **AI 降本与防Token超载策略**：
    - **不传全量正文**：仅将“统计聚合数据（如各情感占比）”和“当日互动量 Top 50 的帖子标题及总结”组装成 Prompt 发给大模型，坚决避免一次性传入数千条长文本。

### 3.6 社媒内容
- **定位**：内容明细检索与溯源。
- **功能**：
  - **核心指标展示**：列表需清晰展示“点赞、收藏、评论、分享”四大关键指标。
  - **高级日期筛选**：支持单日或日期范围查询。
  - **主帖与评论分离架构**：
    - **主帖 Tab**：展示正文、链接、发帖人及指标。
    - **高赞评论 Tab**：将有价值的评论与水评区分开。通过设置阈值（如：字数 > 10 且 点赞数 > 3）筛选高赞评论。展示评论内容、评论指标，并提供**所属主帖的跳转链接**。

### 3.7 KOL发现
- **定位**：发掘优质社区创作者与潜力素人。
- **计算逻辑**：
  - 提取高互动率（如：`(转+评+赞+藏)/预估浏览量 > 5%`）的账号。
  - 结合发帖频率和粉丝数，划分为“头部达人、腰部达人、尾部达人、潜力KOC”。
- **展示信息**：展示该账号近期发帖数、平均互动量、代表作链接，辅助运营制定投放或合作策略。

### 3.8 风控预警
- **定位**：重大负面事件的早期扑灭。
- **预警判定逻辑**：
  - **相对增量触发**：不单纯看绝对值。若某一类负面主题的帖子数量在当日（或近 N 小时）显著提升（如：大于过去 7 天该主题日均值的 1.5 倍），即判定为异常爆发。
- **功能**：
  - 记录预警触发时间、爆发分类。
  - 列出导致该预警的核心高危帖子及负面评论列表。

### 3.9 产品配置
- **定位**：多产品/竞品的底层隔离配置中心。
- **功能**：
  - 支持创建与切换产品（如：鹅鸭杀、竞品B）。
  - **独立生效**：A产品下配置的“监控关键词”、“过滤词”、“自定义话题体系”保存后仅对A生效。

### 3.10 数据更新与数据库管理
- **数据更新**：包含小红书爬虫的启停开关、历史抓取补漏、爬虫频率（默认4小时）设置、爬取日志查看。
- **数据库管理**：支持底层表的直接 SQL 查询验证与脏数据清理。

## 4. 多渠道通用数据库设计方案补充

为支撑以上复杂的运营分析，数据库结构需进一步细化：

### 4.1 核心通用表设计 (`community_contents`)
| 字段名 | 类型 | 说明 | 平台差异说明 |
|---|---|---|---|
| `id` | VARCHAR | 全局唯一ID | 组合键：`{channel}_{original_id}` |
| `project_id` | INT | **关键：区分不同产品配置** | 如：1代表鹅鸭杀 |
| `channel` | VARCHAR | 渠道标识 | `taptap` 或 `xiaohongshu` |
| `content_type` | VARCHAR | 内容载体类型 | `text`(纯文本), `image`(图文), `video`(视频) |
| `post_type` | VARCHAR | 内容层级类型 | **`post`(主帖), `comment`(评论)** |
| `original_id` | VARCHAR | 原平台内容ID | Tap评价ID或XHS帖子/评论ID |
| `parent_id` | VARCHAR | 父级ID | 若为评论，则记录关联的主帖ID |
| `content` | TEXT | 正文内容 | |
| `publish_time` | TIMESTAMP | 发布时间 | |
| `interact_likes` | INT | 点赞数 | |
| `interact_comments` | INT | 评论/回复数 | |
| `interact_shares` | INT | 转发/收藏数 | XHS核心指标 |
| `interact_collects`| INT | 收藏数 | XHS核心指标 |
| `url` | VARCHAR | 原始链接 | |

### 4.2 话题与热度宽表 (`topic_hotspots`)
用于存储计算后的热点数据，避免每次实时全表计算：
- `date` (日期)
- `project_id` (关联产品)
- `topic_name` (话题名)
- `heat_score` (热度值)
- `post_count` (关联帖子数)
- `total_engagement` (总互动量)

### 4.3 动态数据更新策略（针对转评赞变动）
针对“A日入库低指标，B日突然火爆”的情况，系统采取以下同步策略：
1. **二次回扫机制**：
   - **48小时高频回扫**：对入库时间在 48 小时内的帖子，每 12 小时进行一次全量互动指标更新。
   - **7天长效回扫**：对入库 2-7 天的帖子，每天回扫一次。
   - **历史定格**：超过 7 天的帖子，除非手动触发，否则不再更新动态指标（此时舆情价值已定格）。
2. **入库时间 vs 发布时间**：
   - 数据库存储精准的 `publish_time`（用户发布时间）。
   - 前端所有日期筛选均基于 `publish_time`，确保运营能准确追踪特定日期发布的内容表现。
   - 记录 `first_crawl_time`（首次入库时间），作为审计参考。

## 5. 后续开发规划路径
1. **重构菜单**：立刻在 `App.vue` 中落实上述 11 个小红书左侧菜单，并将“使用指南/简介”移至全局右上角。
2. **表结构迁移**：利用 `DatabaseManage` 或后端脚本，应用 `community_contents` 表及 `project_id` 逻辑。
3. **分模块实现视图**：按照运营视角的逻辑，从“实时舆情”、“社媒内容”开始，逐个实现上述定义的图表与数据过滤能力。
