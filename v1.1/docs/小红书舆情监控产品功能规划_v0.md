# 小红书舆情监控产品功能规划 v0

## 1. 本轮结论与边界

- 目标：在现有 TapTap 舆情平台 v1.1 上，新增小红书舆情监控能力，并与 TapTap 形成统一分析入口。
- 范围：本轮先做规划与落地路线，先复用 MediaCrawler 的小红书抓取链路，数据库继续使用现有 PostgreSQL。
- 原则：先跑通、后优化；先统一数据主干、后做平台特化能力；先保障账号安全、后追求抓取覆盖率。

## 3. 产品功能清单（先抄版，后优化）

以下为可直接进入产品需求池的“先抄版”能力清单，先完整落地，再做优先级优化。

### 3.1 数据来源与语料覆盖

- 海内外主流媒体、社区与私域的来自官方、KOL 及普通玩家的真实声音。
- 小红书侧当前先落地公开内容采集：帖子、评论、互动数据、作者基础信息。

### 3.2 GenAI 语料处理能力

- 对全量语料做情感分析。
- 对全量语料做关键词提取。
- 对全量语料做重要度评估。
- 对全量语料做风控识别。

### 3.3 热点发现与追踪解读

- 基于 GenAI 自主发现社区热点话题。
- 支持热点话题追踪与定期解读。
- 支持按主题输出“趋势变化 + 风险摘要 + 运营建议”。

### 3.4 业务价值导向

- 结合游戏内行为数据进行联动分析。
- 支持运营决策：问题优先级、修复反馈追踪、版本口碑复盘。

## 4. 与 TapTap 的一体化设计

## 4.1 代码层：合并还是分开

- 结论：采用“业务层合并、平台采集层分开”的混合方案。
- 平台采集层分开：
  - TapTap 保持现有抓取服务。
  - 小红书通过 `MediaCrawler Adapter` 接入。
  - 原因：平台登录态、反爬策略、字段结构差异大，强行合并会让采集层脆弱。
- 业务层合并：
  - 统一分析服务、统一告警服务、统一查询接口。
  - 原因：情感/主题/摘要/告警逻辑跨平台可复用，便于趋势对比和多平台看板。

## 4.2 前端展示层：可照抄与必须新增

- 可照抄 TapTap 的：
  - 情感分布（正向/负向/中性）
  - 主题分类（问题分类）
  - 一句话总结
  - 负面舆情监控页
  - 词云分析页
  - 舆情分析报告页
- 小红书必须新增的：
  - 互动指标面板：点赞、收藏、评论、分享
  - 内容类型筛选：图文/视频、帖子/评论
  - 作者维度：KOL 发现、账号影响力分层
  - 平台特有风控信号：异常删帖、评论异常波动、账号状态异常
  - 检索策略可视化：按时间、热度、综合的抓取覆盖看板

## 4.3 后端服务分层建议

- `adapters/platforms/taptap`：现有 TapTap 采集适配。
- `adapters/platforms/xhs`：新增 MediaCrawler 适配器。
- `services/unified_ingest_service.py`：统一清洗与入库。
- `services/unified_analysis_service.py`：统一情感、主题、摘要、风险识别。
- `services/unified_alert_service.py`：统一预警引擎（阈值、突增、连续性异常）。
- `api` 统一对外：`platform=taptap|xhs|all`。

## 5. 数据库方案（复用现有 PostgreSQL）

- 保留现有核心表：`platforms / products / reviews / crawl_logs`。
- 新增平台记录：在 `platforms` 增加 `xiaohongshu`。
- 在 `reviews` 上做扩展字段：
  - `source_type`（note/comment/review）
  - `content_id`（帖子或评论主键）
  - `parent_content_id`（评论挂载帖子）
  - `author_id`
  - `content_url`
  - `engagement_json`（赞藏评转等）
  - `extra_json`（平台特有原始字段）
- 去重策略：
  - 主去重：`platform_id + review_id`（沿用）
  - 兜底去重：`platform_id + content_hash + review_date + author_id`
- 支持竞品监控：
  - 复用 `products` 表，通过产品开关控制是否抓取与告警。

## 6. 小红书反爬与账号安全策略

- 账号策略：
  - 生产与测试账号隔离，主账号不用于高频采集。
  - 保留登录态，优先复用稳定会话，减少反复扫码。
- 行为策略：
  - 降低动作密度：随机停顿、分批抓取、错峰执行。
  - 禁止高频深翻页与高并发详情点击。
- 流量策略：
  - 低并发（默认 1）+ 速率限制。
  - 必要时接入代理池并控制轮换频率，避免异常突变。
- 采集策略：
  - 先抓卡片层元数据，命中规则后再抓正文与评论。
  - 对“低价值水帖”做早筛，减少深抓动作总量。
- 风险监控：
  - 新增反爬风险状态：登录失效、验证触发、异常跳转、请求失败率突增。
  - 命中风险阈值自动降频或暂停任务，触发人工介入。

## 7. 小红书抓取策略与实验设计

## 7.1 每日全面抓取是否可行

- 不建议“全量扫全站”，账号风险高且水帖占比高。
- 建议“分层抓取”：
  - 第一层：搜索结果卡片快速采样（低成本）。
  - 第二层：高价值帖子深抓（正文+评论）。
  - 第三层：重点主题持续跟踪（增量抓取）。

## 7.2 搜索筛选如何抓

- 关键词：主词 + 近义词 + 竞品词包（鹅鸭杀、鹅鸭杀手游、Goose Goose Duck 等）。
- 维度组合：
  - 时间维度：近 1 天 / 近 7 天
  - 排序维度：综合 / 点赞高 / 评论高
- 执行建议：
  - 每日任务拆成多个“低频小批次任务”，避免单次超量。
  - 同关键词不同筛选交叉取样，最后做去重与融合排序。

## 7.3 预警场景下怎么抓

- 目标预警：近 N 小时负面主题帖子暴增。
- 采集节奏：
  - 常态：低频巡检（例如每 2-4 小时）。
  - 告警态：针对命中主题短时加密抓取（仅该主题与相关关键词）。
- 判定规则（v0）：
  - 当前窗口负面量较历史基线提升超过阈值。
  - 同时满足互动量门槛（避免被低质噪声触发）。
  - 持续两个窗口仍异常才升级告警等级。

## 8. 功能扩展路线（与现有规划对齐）

- 舆情监控与告警：多平台统一阈值 + 平台特化阈值。
- 舆情分析：跨平台情感、主题、版本前后对比。
- 社区内容分析：帖子结构、互动结构、话题扩散路径。
- 版本口碑分析：版本切片、发布时间窗、口碑拐点。
- KOL 发现：作者分层、话题影响力、情感倾向。
- 竞品社区监控与运营策略分析：
  - 产品维度开关启用。
  - 支持同关键词在竞品产品集合下独立追踪。

## 9. 分阶段落地建议

- Phase 1（现在）：跑通小红书抓取 + 入库 + 基础展示。
- Phase 2：统一查询接口 + 平台切换视图 + 基础预警。
- Phase 3：KOL、竞品、版本口碑、深度专题分析。
- Phase 4：热点自动解读、跨平台联动报告自动化。
