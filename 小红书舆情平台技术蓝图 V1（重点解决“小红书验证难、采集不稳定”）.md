设计原则

- 目标不是“永远不触发验证”，而是“触发验证后系统仍可持续产出”。
- 严格合规采集：不做破解类方案，只做会话管理、限频、降级恢复。
- 先保核心监控面（官号/竞品号/重点帖子），再扩全网召回。
- 沿用你现有架构：FastAPI + APScheduler + Vue 大屏。
一、系统架构 V1

- 采集入口分三类：账号订阅流（稳定）、关键词搜索流（波动）、帖子回查流（高价值）。
- 中间层新增“会话中心 + 风控状态机 + 任务队列”。
- 下游复用你现有分析链：情感、主题、摘要、风险标签。
- 前端新增“采集健康页”，和已有大屏并列展示。
- 可复用后端入口与调度模式参考 main.py 与 scheduler.py 。
二、数据库设计（新增）

- xhs_sessions ：会话池（session_id, account_id, status, last_validated_at, expire_at, risk_level）。
- xhs_crawl_tasks ：任务队列（task_type, priority, status, retry_count, next_run_at, fail_reason）。
- xhs_notes ：笔记主表（note_id, author_id, title, content, publish_time, metrics_json, raw_json）。
- xhs_note_comments ：评论增量表（comment_id, note_id, content, like_count, created_at）。
- xhs_risk_events ：风险事件表（event_type, severity, note_id, trigger_rule, detected_at）。
- 命名风格对齐你当前模型方式，可参考 review.py 和 crawl_log.py 。
三、任务状态机（关键）

- 任务状态： pending -> running -> success/failed/risk_blocked/manual_required 。
- 会话状态： healthy -> warning -> blocked -> reauth_required -> healthy 。
- 触发验证码时：任务转 manual_required ，并自动降级到“账号订阅流继续跑”。
- 连续失败 N 次：转 risk_blocked ，进入冷却窗口，避免账号进一步失活。
- 人工续期完成后：状态机自动回切，不需要改代码。
四、降级恢复机制

- 一级降级 ：停止关键词流，仅保账号订阅流与帖子回查流。
- 二级降级 ：仅抓重点白名单（官号、竞品官号、重点KOL）。
- 三级降级 ：仅做历史数据分析与告警，不做新抓取。
- 所有降级自动记录事件，并在大屏显示“采集覆盖率”变化。
五、后端接口清单（V1）

- POST /api/xhs/sessions/refresh ：人工续期后更新会话状态。
- GET /api/xhs/sessions/status ：会话池健康度（可做系统监控卡片）。
- POST /api/xhs/tasks/dispatch ：下发采集任务（类型、优先级、范围）。
- GET /api/xhs/tasks ：任务列表与失败原因。
- POST /api/xhs/tasks/{id}/retry ：人工触发重试。
- GET /api/xhs/health/overview ：成功率、验证触发率、覆盖率、数据新鲜度。
- GET /api/xhs/insight/version ：版本窗口主题、槽点、风险变化。
六、前端页面（沿用现框架）

- 新增 XhsHealth.vue ：会话健康、任务成功率、验证触发趋势、降级状态。
- 新增 XhsInsight.vue ：版本话题聚类、槽点TOP、竞品节奏对比。
- 复用你现在的路由与容器组织方式，可参考 App.vue 和 SystemMonitor.vue 。
七、实施里程碑（建议）

- 里程碑1：会话中心 + 任务队列 + 健康监控 API 打通。
- 里程碑2：账号订阅流稳定运行，支持人工续期与自动回切。
- 里程碑3：关键词流接入并实现降级策略。
- 里程碑4：版本洞察页 + 风险告警页上线。
- 里程碑5：竞品节奏分析（官号/玩家号/三方号）完成并自动周报。
八、验收指标（必须量化）

- 采集成功率（非风控异常）≥ 95%。
- 会话失效率可观测且可恢复（人工续期后自动恢复链路）。
- 关键账号覆盖率 ≥ 99%（白名单集）。
- 版本窗口热点识别延迟 ≤ 30 分钟。
- 风险告警误报率持续下降（每周人工复盘校准）。
